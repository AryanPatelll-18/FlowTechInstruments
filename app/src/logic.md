# System Logic

This document describes the document-processing flow in `app/src`, focused on how uploaded files are handled, converted to HTML, and then printed or saved as PDF.

## 1. File Intake

The main entry point for document upload is `DatasheetGeneratorPanel`.

- The user can either paste SO / quotation text or upload a file.
- Accepted upload types in the UI are `.pdf`, `.txt`, `.csv`, `.doc`, and `.docx`.
- The upload handler reads the selected file and converts it to plain text before parsing.
- The current reader implementation directly supports PDF, TXT, and CSV. The `.doc` and `.docx` options are accepted in the picker, but they are not handled by the text extractor itself.

## 2. Text Extraction From Uploaded Files

Uploaded files are normalized into text in `app/src/data/pdfExtractor.ts`.

- If the file is a PDF, the app uses `pdfjs-dist` to extract text from each page.
- The extractor preserves line order and tries to keep table-like structure by inserting spacing markers when text chunks are separated by larger gaps.
- If the PDF contains no selectable text, the file is treated as scanned and parsing is rejected with an error message.
- If the file is plain text or CSV, it is read with `FileReader.readAsText`.

Important detail:

- The current extraction flow is text-based.
- The actual datasheet builder works from extracted text, not from binary file contents.

## 3. Parsing Uploaded Content

The extracted text is passed to `generateFromSOText()` in `app/src/data/api.ts`.

That function does three things:

1. Calls `parseSOText()` from `app/src/data/dataMapper.ts`.
2. Tries to auto-match GA drawings for each detected line item.
3. Generates the final HTML datasheet.

### 3.1 Structured data extraction

`parseSOText()` converts raw quotation text into a structured `SODatasheet` object:

- It delegates to `parseSmart()` in `app/src/data/smartParser.ts`.
- The parsed result is split into:
  - a document header
  - one `LineItem` per instrument
  - section rows for each line item
- The parser also collects notes and stores them in the header.

### 3.2 GA drawing lookup

After parsing, `generateFromSOText()` attempts to find a matching GA drawing for each line item.

- It calls `findGADrawing()` from `app/src/data/gaDrawingLookup.ts`.
- Matching is based on product family, instrument type, size, and de-codification number.
- If a match is found, the line item gets:
  - `gaDrawingUrl`
  - `gaDrawingTotal`

This means the datasheet can embed the correct drawing automatically if the drawing exists in Document Master.

## 4. HTML Generation

The HTML output is generated by `generateDatasheetTemplate()` in `app/src/data/datasheetTemplate.ts`.

The template renders a complete technical datasheet as a single HTML document.

### 4.1 What the HTML contains

- A cover page
- An instrument index page
- One page per instrument / line item
- A closing standards or reference section

### 4.2 How each instrument is rendered

For every parsed line item, the template builds:

- A header bar with project and order information
- A compact specification table built from all parsed sections
- A GA drawing panel
- A connection-dimension table when a process connection is detected

### 4.3 How GA drawings appear in the HTML

- If a GA drawing URL exists, the template inserts it directly.
- If the drawing is an image, it renders with `<img>`.
- If the drawing is a PDF, it renders with `<iframe>`.
- If no drawing is available, a placeholder box is shown instead.

## 5. Previewing The HTML

The generated HTML is previewed in the browser inside an `<iframe>`.

In `DatasheetGeneratorPanel`:

- `setHtml(r.html)` stores the generated HTML.
- The preview tab uses `iframe srcDoc={html}` to render the full datasheet.
- Editing any header or section field regenerates the HTML immediately by calling `generateFromDatasheet()`.

## 6. Converting HTML To PDF

The app does not create PDFs on the server.

Instead, it uses the browser print pipeline:

- `openPrintWindow(html)` in `app/src/data/api.ts` opens a new browser window, writes the HTML into it, and triggers `window.print()`.
- `DocumentViewer` uses the same pattern inside an iframe and calls `contentWindow.print()`.

When the user chooses **Print / Save PDF** or **Print PDF**:

- The HTML is rendered in a print-friendly layout.
- The browser print dialog opens.
- The user saves the output as a PDF from the print dialog.

So the "HTML to PDF" step is actually:

1. Generate HTML.
2. Load HTML into a print context.
3. Use the browser's print function.
4. Save as PDF through the print dialog.

## 7. Data Flow Summary

The core flow is:

1. User uploads a file or pastes SO text.
2. The file is converted to plain text.
3. The text is parsed into structured header + line items.
4. GA drawings are auto-matched when available.
5. The structured data is rendered into HTML.
6. The HTML is previewed in an iframe.
7. The browser print dialog converts that HTML into a PDF output.

## 8. Practical Notes

- PDF extraction works only when the PDF contains selectable text.
- Image-based scanned PDFs are rejected by design because they cannot be reliably parsed as text.
- The PDF output depends on browser print behavior, so the final appearance is controlled by the HTML and print CSS.
- GA drawing insertion is optional but automatic when matching data exists in Document Master.
